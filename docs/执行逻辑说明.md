# 当前用例执行逻辑说明

## 一、整体流程概览

```
启动测试 (npm run test / npx playwright test)
    │
    ├─ 1. 读取 playwright.config.ts
    ├─ 2. 决定是否注入登录态 (auth.json)
    ├─ 3. 启动浏览器（Chromium），应用全局 use 配置
    ├─ 4. 对每个 *.spec.ts 中的每个 test()：
    │       ├─ 若用例用了 fixtures/base 的 test → 先跑 loggedInPage fixture
    │       └─ 再跑用例本体
    └─ 5. 输出报告（本地 html / CI 时额外 junit.xml）
```

---

## 二、配置层（playwright.config.ts）

| 项 | 作用 |
|----|------|
| **testDir** | 用例目录 `./tests` |
| **testMatch** | 只执行 `**/*.spec.ts`，不会跑其他 .ts |
| **baseURL** | `https://365.kdocs.cn/kmwiki`，用例里 `page.goto('/')` 即打开该地址 |
| **storageState** | 若存在 `auth.json`（或环境变量 AUTH_FILE 指向的文件），则**每个用例**启动时自动注入其中的 Cookie/Storage，相当于“已登录” |
| **use** | 所有用例共用：有头、1980×1080、中文、截图/录像/失败时 trace 等 |

因此：

- **有 auth.json**：每个用例一开始就是在“已登录”的浏览器上下文里跑。
- **没有 auth.json**：不注入登录态，需要登录的用例会处于未登录状态（可能失败）。

---

## 三、用例的两种写法（决定“有没有先打开首页”）

### 类型 A：用 fixture，先打开知识库首页再操作

**代表**：`test-1.spec.ts`、`demo.spec.ts`

- 从 **`./fixtures/base`** 引入 `test`（不是直接从 `@playwright/test`）。
- 用例参数用 **`loggedInPage`**（或 `loggedInPage: page`）。
- **执行顺序**：
  1. 配置的 storageState 已注入（若有 auth.json）。
  2. **Fixture**：先 `page.goto('/')`（即 baseURL），再 `waitForLoadState('networkidle')`，相当于“已登录且停在知识库首页”。
  3. 用例里直接写从首页开始的操作（如点「团队文档」、智能问答等）。

适合：**从知识库首页开始**的录制/手写用例。

### 类型 B：用默认 test，自己 goto 目标页

**代表**：`self.exam.spec.ts`、`example.spec.ts`、`test-3.spec.ts`

- 从 **`@playwright/test`** 引入 `test`。
- 用例参数用 **`page`**。
- **执行顺序**：
  1. 配置的 storageState 已注入（若有 auth.json）。
  2. **没有**“先打开首页”的步骤，由用例自己决定去哪。
  3. 用例里第一步通常是 `page.goto('某个 URL')`（如测验页、百度等）。

适合：**从任意 URL 开始**的用例（例如直接打开测验页、或外站）。

---

## 四、单条用例从开始到结束在做什么

以 **self.exam.spec.ts** 为例：

1. Playwright 为该用例创建一个浏览器上下文。
2. 若存在 auth.json，把其中的 Cookie/Storage 写入该上下文 → **已是登录态**。
3. 用例运行：`page.goto('https://www.kdocs.cn/wiki/.../qa/')`，打开测验页（带登录态）。
4. 后续步骤：选文档、生成试题、答题、导出等。

以 **test-1.spec.ts** 为例：

1. 同样先注入 auth.json（若有）。
2. **Fixture** 执行：`page.goto('/')` 打开知识库首页，并等待 networkidle。
3. 用例运行：从首页点「团队文档」→ 选库 → 智能问答等。

---

## 五、登录态从哪里来（和用例执行的关系）

- 用例**不负责登录**，只负责业务步骤。
- 登录态由 **auth.json** 提供；auth.json 由 **`npm run auth`**（脚本 `scripts/save-auth.js`）生成。
- 脚本用 **.env 里的 WPS_SID** 打开浏览器，访问 365 知识库，然后把当前会话保存为 auth.json。
- 因此执行逻辑是：**先跑一次 `npm run auth` 生成 auth.json → 之后所有用例执行时自动带登录态**。

---

## 六、小结表

| 用例文件 | 用的 test | 是否先打开首页 | 是否依赖登录 |
|----------|-----------|----------------|--------------|
| test-1.spec.ts | fixtures/base | 是（loggedInPage） | 是（知识库） |
| demo.spec.ts | fixtures/base | 是 | 是 |
| self.exam.spec.ts | @playwright/test | 否，自己 goto 测验页 | 是（同一 Cookie 域） |
| test-3.spec.ts | @playwright/test | 否，goto 百度 | 否 |
| example.spec.ts | @playwright/test | 否，goto playwright.dev | 否 |
| test-2/4/5/6/7 | @playwright/test | - | test.skip 不执行 |

整体上：**配置统一注入 auth.json → 需要从首页开始的用 fixture 打开首页 → 其余用例自己 goto；登录只依赖 auth.json，不写在用例里。**
